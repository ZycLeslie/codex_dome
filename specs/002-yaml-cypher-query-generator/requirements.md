# 需求说明

## 背景

需要提供一个独立、通用的 Java 模块，接收 YAML 形式的图查询定义，并生成可直接交给图库驱动执行的参数化 Cypher 语句与参数表。该能力要覆盖线性路径、共享节点形成的分叉路径，以及节点/边的过滤条件。

## 目标

提供一个基于 YAML 接口定义的图查询生成器，支持：

- 描述节点、边、路径与分支
- 按 `id` 与属性过滤节点和边
- 在边类型未知时仍生成合法 Cypher
- 指定返回目标与结果数量上限
- 输出生产可用的参数化 Cypher 与参数 `Map`

## 功能需求

### FR-1 YAML 查询文档

系统必须支持解析一个结构化 YAML 文档，至少包含：

- `version`
- `query.nodes`
- `query.edges`
- `query.paths`
- `query.return`
- `query.limit`

其中：

- `nodes` 以节点别名为 key，描述节点标签与过滤条件
- `edges` 以边别名为 key，描述边两端节点、方向、类型与过滤条件
- `paths` 通过边别名数组描述一条连续路径

### FR-2 路径与分支表达

系统必须支持：

- 单链路路径，例如 `A-B-C`
- 共享节点或共享前缀形成的分支，例如 `A-B-C` 与 `B-D`，或 `A-B-C` 与 `A-B-D`

分支通过多个命名路径共享同一节点或边别名来表达。

### FR-3 节点过滤

每个节点定义必须支持以下过滤能力：

- `id`
- `properties`

过滤值必须在生成的 Cypher 中以参数形式出现，不能直接拼接到值位置。

### FR-4 边过滤

每个边定义必须支持以下过滤能力：

- `id`
- `type`
- `properties`
- `direction`

其中：

- `type` 可选，未提供时仍需生成合法边模式
- `direction` 至少支持 `OUTBOUND`、`INBOUND`、`UNDIRECTED`

### FR-5 返回目标

系统必须支持通过 YAML 明确声明返回项。返回项至少支持：

- `PATH`
- `NODE`
- `EDGE`

每个返回项必须支持：

- 引用已定义的路径/节点/边别名
- 为结果列声明输出别名

若未声明返回项，系统应默认返回全部命名路径。

### FR-6 limit

系统必须支持在 YAML 中声明结果条数限制，并在生成 Cypher 时带出 `LIMIT`。

### FR-7 Java API

系统必须提供清晰分层的 Java API，至少包含：

- YAML 对应 DTO / 配置对象
- YAML 解析入口
- 校验入口
- Cypher 生成入口
- 输出对象，包含 `query` 与 `params`

### FR-8 明确错误处理

系统必须对以下错误提供清晰异常：

- YAML 结构错误
- 缺失必要字段
- 非法标识符
- 路径不连续
- 路径引用未知节点/边
- 返回项引用未知对象
- `limit` 非法

## 非功能需求

### NFR-1 独立可复用

实现应作为独立 Java 模块存在，不依赖当前 Chrome 扩展实现。

### NFR-2 安全性

除 Cypher 结构位必须使用的标识符外，所有过滤值必须参数化，避免注入风险。

### NFR-3 可维护性

实现需分离解析、校验、模型编译、Cypher 生成职责，不允许把全部逻辑堆在单一类中。

### NFR-4 可测试性

需要提供基础自检能力，至少覆盖：

- 单链路生成
- 分叉生成
- 边类型缺省
- 节点/边 `id + properties` 过滤
- 非法 YAML 或非法路径校验

## v1 不支持范围

以下内容不属于本次范围：

- 任意 Cypher 片段透传
- 可变长路径
- `OR` / `NOT` / 比较运算组合
- 内联 YAML 集合语法、锚点、别名、merge key 等高级 YAML 特性
- 对 Neo4j 内部 `elementId()` / `id()` 函数语义做特化支持

`id` 在本功能中统一表示业务属性 `id`，即生成 `alias.id = $param` 形式的过滤。
