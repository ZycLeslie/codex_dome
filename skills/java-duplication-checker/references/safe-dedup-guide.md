# Safe Dedup Guide (Java)

## 目标

在不改变行为的前提下减少重复代码。去重必须可回滚、可验证、可审计。

## 策略矩阵

| 场景 | 优先策略 | 暂缓策略 |
| --- | --- | --- |
| 同一类内重复 `private` 方法 | 保留一个实现，其余改委托后删除 | 直接改 public API |
| 同一类内重复 `public/protected` 方法 | 抽 `private helper`，原方法保留签名 | 合并成一个新 API 再改所有调用 |
| 多类间低风险重复（静态/纯计算） | 提取共享 utility/static helper，原方法短期委托 | 一步删除所有旧入口 |
| 多类间中风险重复（依赖实例状态） | 提取组合式 helper/service，通过注入调用 | 强行提升为继承体系 |
| 高风险重复（事务/并发/外部 IO） | 先补表征测试和隔离层，再小步迁移 | 直接抽取并删除旧实现 |

## 必做校验

1. public/protected 方法签名与可见性保持不变。
2. 注解语义保持不变：事务、安全、缓存、重试、异步。
3. 异常契约保持不变：异常类型、抛出时机、错误码。
4. 副作用顺序保持不变：DB/消息/HTTP/文件写入顺序。
5. 先通过编译，再通过受影响单元测试和集成测试。
6. 每次只改一个重复组，避免混合变更掩盖回归问题。

## 高风险信号

- `@Transactional` / `@Async` / `@Cacheable` / `@PreAuthorize`
- `synchronized`、锁对象、线程同步
- `this.field = ...` 或集合写操作（`add/remove/put`）
- 外部调用：repository、JDBC、HTTP、消息队列、文件系统
- `throw/throws` 语义敏感分支

当任一信号出现时，默认按高风险处理：

1. 不直接合并实现。
2. 先加表征测试锁定行为。
3. 用委托桥接逐步替换调用点。
4. 分阶段删除旧实现。

## 推荐落地顺序

1. 处理“同类内 private 完全重复”。
2. 处理“同类内 public/protected 重复”。
3. 处理“跨类低风险重复”。
4. 最后处理“跨类中高风险重复”。

## 失败回滚建议

- 每个重复组单独 commit，便于精确回滚。
- 保留短期兼容入口（委托层）直到验证通过。
- 若出现行为偏差，优先回滚当前重复组，而非整批重构。
