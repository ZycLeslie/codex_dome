# Protected Root Patterns

Load this reference when dead-code candidates touch framework entry points, generated APIs, or configuration-driven dispatch.

## Java and JVM

Protect methods/classes with or implementing:

- Spring MVC/WebFlux: `@Controller`, `@RestController`, `@RequestMapping`, `@GetMapping`, `@PostMapping`, `@PutMapping`, `@DeleteMapping`, `@PatchMapping`, router functions.
- Generated or handwritten API interfaces: `*Api`, `*Controller`, `*Resource`, classes implementing OpenAPI-generated interfaces, methods matching `operationId`.
- Spring lifecycle and DI: `@Bean`, `@Configuration`, `@Component`, `@Service`, `@Repository`, `@Autowired`, `@PostConstruct`, `@PreDestroy`, `InitializingBean`, `DisposableBean`.
- Async/events/jobs: `@Scheduled`, `@Async`, `@EventListener`, `ApplicationListener`, `CommandLineRunner`, `ApplicationRunner`.
- Messaging: `@KafkaListener`, `@RabbitListener`, JMS listeners, stream bindings, consumer methods referenced by topic/group config.
- Persistence/serialization: JPA entity callbacks, converters, Jackson `@JsonCreator`/`@JsonValue`/custom serializers, MapStruct mappers, validation annotations.
- Reflection/config: class names, bean names, method names, route paths, or handler names in `application.yml`, `application.properties`, XML, migration scripts, and shell scripts.

## YAML or JSON Interface Specs

Protect source matching any of these spec signals:

- OpenAPI/Swagger `paths`, HTTP methods, `operationId`, tags, `x-controller`, `x-handler`, `x-service`, `$ref`.
- AsyncAPI `channels`, operations, message handlers, consumer names, bindings.
- Gateway or route configs with path predicates, upstream service names, controller/handler references.
- RPC descriptors containing service, interface, method, or operation names.

Mapping rules:

- Match `operationId` to method names exactly first, then camel/snake/kebab variants.
- Match path templates such as `/users/{id}` to route annotations and constants.
- For generated Java APIs, protect both the generated interface method and every implementation/override.
- If a spec operation is present but no implementation is found, keep likely matching controllers/services until a human or test proves they are obsolete.

## JavaScript and TypeScript

Protect route handlers and framework entry points:

- Express/Koa/Fastify route registrations and middleware.
- NestJS controllers, providers, guards, interceptors, decorators, and module providers.
- Next.js/Remix/SvelteKit route files, loaders/actions, API routes, server actions.
- React/Vue/Svelte components referenced by routing tables, dynamic imports, templates, or string-based component registries.
- Package exports, CLI bins, plugin hooks, worker handlers, cron jobs, and queue consumers.

## Python

Protect:

- FastAPI/Flask/Django routes, views, serializers, model methods used by frameworks, management commands.
- Celery/RQ/Arq tasks, signal handlers, startup/shutdown hooks, plugin entry points.
- Symbols referenced in settings, URL configs, templates, migrations, or import strings.

## General Keep Rules

Keep or mark uncertain when evidence depends on:

- Reflection, dynamic imports, eval, classpath scanning, plugin loading, service discovery, or dependency injection by string.
- Public APIs of a package/library without a known full consumer set.
- Generated code that will be regenerated by build tooling.
- Test-only callers when the user wants to preserve test helpers; otherwise treat test-only reachability as non-production evidence and state that assumption.
