# Project Delivery Standard

这份文档提供“项目级”交付门禁默认值。若仓库本身存在更严格的 `README`、`docs/`、`specs/`、`openspec/changes/` 约束，优先使用仓库文档覆盖这里的默认规则。

## 1. 先规格，后实现

开始实现前，先读完整设计链：

- 新工作优先使用 OpenSpec：`proposal.md` / `spec.md` -> `design.md` -> `tasks.md`
- 旧 `specs/` 只在 feature 尚未迁移或用户明确指定时作为迁移源使用
- 若 OpenSpec 和旧 `specs/` 同时存在且冲突，优先 OpenSpec；若业务语义不清，先确认再编码

若规格缺失、冲突、过期或和当前实现明显不一致：

- 先补规格或对齐边界
- 再进入代码实现

开始编码前，再回答这 4 个问题：

1. 这次改动是否改变了用户能力？
2. 这次改动是否改变了模块职责或数据流？
3. 这次改动是否引入了新状态、配置项或错误策略？
4. 当前 `tasks.md` 是否已经覆盖这次工作？

若任一问题答案明显为“是”或“未覆盖”，先更新规格和任务文档。

## 2. 文档与实现分层

- 需求和设计放在 `docs/`、`specs/`、`openspec/changes/`
- 真正实现只落在 `src/` 或目标模块目录
- 不要把设计说明堆进源码文件
- 不要在根目录散落临时说明文档

## 3. 任务闭环

- 在开始实现前，先建立本地清单
- 在开始编码前，把本轮计划写进 `tasks.md`、OpenSpec `tasks.md` 或等价任务记录
- 每个执行切片完成后，必须更新清单状态
- 若仓库已有 `tasks.md` 或 OpenSpec 任务文档，需要在必要时同步完成项和剩余项
- 不要只在聊天回复里说“计划做什么”，本地记录必须同步

## 4. 变更边界

- 不要绕过设计文档直接做大改
- 不要混入与当前需求无关的重构
- 每轮只推进一个可验证闭环
- 若发现额外问题，优先记录风险，不要顺手扩展需求

## 5. 完成定义

一个实现任务只在以下条件全部满足时才算完成：

1. 相关设计与任务文档已对齐
2. 代码已完成最小可用闭环
3. 已完成最小必要自检或构建检查
4. 本地清单全部为 `DONE + OK`
5. 必要文档已同步

## 6. 本 Skill 的默认额外约束

- 默认不考虑测试
- 仍应尽量保留构建可通过状态
- 若用户额外要求测试，再把测试加入任务单和最终门禁
